### 選定理由

#### GCP Cloud Run
社内既存案件は PHP + AWS EC2 構成で運用されており、本プロジェクトの GCP Cloud Run + Go + distroless 構成と費用比較した結果、約10倍の費用効率を実現した。
設定の認知負荷も低く、Go バイナリ ＋ distroless（OS なし）イメージとの相性が非常に良い点も決め手となった。

#### GCP Cloud SQL（PostgreSQL 18）
PostgreSQL は MySQL に比べて unsigned 型や 1byte 型がなくデータ効率の面では劣る部分があるが、拡張機能による BIG データ操作・計算系処理の豊富さが今回の要件と合致したため選定された。

#### sqlc
sqlをコンパイラして静的なクエリをアプリケーション層から関数として安全に呼び出せるものである。
ORMよりpostgreの機能を最大限使えるためSQLと同等の速度と自由度かつより安全なsqlcが選定された。
pgxにより更に高速化されている。

#### Go 1.26 (Echo v5)
RestAPI + 利用者ポータルで大量のトラフィックを捌かなければならないため、軽量なゴルーチンとシングルバイナリで動く Go を選定した。
チーム開発にも Go は有用で、認知負荷が低く simple than clever が推奨される言語特性により、誰が書いても読みやすさが一定に保たれる。この点も評価され、対抗であった Rust を抑えて Go が選ばれた。
バージョンは本番リリースが 2026 年 5 月であり、それまでに最新 stable に追従して検証する余裕があったため、Go 1.26 / Echo v5 を先行採用している。

#### gRPC
社内のマイクロサービス間の通信には[AIP](https://google.aip.dev/)に従いgRPCを用いる。Http2によるファイルの高速転送や、型情報を載せたまま情報を送ることができる。
各チームの技術レベルに合わせ、JsonによるRestAPIもAIPに従って公開されている。

#### Inertia
利用者ポータルのフロントとバックの通信の認知負荷を減らすことのできるライブラリで、デメリットがないため導入。
Vue,React,Svelteなどの主要フロントエンドライブラリに対応している。

#### Svelte5
Svelte4よりも認知負荷が上がってしまいReact感が強くなったが、仮想DOMなしで動くReactiveなフレームワークで軽量高速なため、モバイルと相性が良いので選定された。
内部のワークフローでtygoによりRequest+Response+enumの型がGoより自動生成される。

#### Claude Code
ワークフローにClaude Codeを組み込むことで、誰が書いてもクオリティがある程度保証される
